### 🛡️ Cyber — Security view

**Purdue / 62443 map**
- S7-1500 on VLAN 20 = Level 1/2 (Basic Control / Area Supervisory), IEC 62443 zone "Pilger Cell".
- OPC UA historian in DMZ = Level 3.5 (iDMZ conduit between OT and Enterprise).
- Current ask: direct Modbus/TCP 502 from L3.5 Linux VM into L1 PLC. That is a **conduit puncturing a zone boundary with a Tier 1 (1979-era) unauth protocol** (see deep-dives/protocol-lineage.md line 22). 62443-3-3 SR 1.1/1.2 (identification & authentication) cannot be satisfied — Modbus TCP has no auth by spec.

**Top 3 silent failure modes**

1. **Stale-but-"Good" tags after TCP half-open.** Modbus TCP carries no timestamp and no quality bit. If the S7 CP connection hangs (keepalive mis-tuned, firewall idle-timeout on DMZ stateful FW kills session after e.g. 3600 s), the OPC UA historian's *bridge* re-serves the last register value with `StatusCode=Good` and a freshly-stamped `SourceTimestamp` generated by the gateway — not the PLC. Operators see "live" pilger mandrel force that hasn't updated in hours. Root cause: the Modbus→UA bridge synthesises quality/timestamp it has no basis for (protocol-lineage.md §"Modbus → OPC UA gateway", line 90).

2. **Byte/word-order flip on 32-bit floats → wrong, not missing.** S7-1500 `REAL` published via Modbus FC03 occupies two 16-bit registers. Siemens default is big-endian word order (ABCD); many Linux OPC UA gateways (kepware-like, Neuron, unified-automation) default to CDAB or BADC. Result: roll-force tag reads 1.7e-38 N or 6.02e23 N instead of 850 kN — historian stores it, trend looks flat-line-then-spike, MES KPI dashboards average garbage (protocol-lineage.md lines 100–101). No alarm fires because the value is numerically valid.

3. **Unit-ID / register overlap + unauth writes that the PLC will happily execute.** Modbus TCP on 502 is unauthenticated (protocol-lineage.md line 22). The "historian" is specified as *subscribe*, but the TCP socket is bidirectional: any process on the DMZ VM — or an attacker who pivots to it via the MES chain — can send FC06/FC16 to the same Unit ID and mutate setpoints. S7-1500 open-user-communication blocks do not enforce read-only semantics on Modbus server FBs unless explicitly coded. Silent because writes succeed with no audit trail on the PLC (no syslog of Modbus writes by default).

**SWOT**
- Strengths: cheap, fast to stand up, no new hardware, MES team gets UA-native subscriptions they already know.
- Weaknesses: cleartext Tier-1 protocol crossing a zone boundary; no auth, no integrity, no timestamp; DMZ VM becomes dual-homed pivot.
- Opportunities: swap PLC-side exposure to **native OPC UA server on S7-1500** (firmware ≥ V2.6) with SecurityPolicy Basic256Sha256 + user tokens — kills Modbus on the wire.
- Threats: 2025 CVEs on S7-1500 firmware v3.1.5 — **CVE-2025-38079 double-free (CVSS 7.8)**, **CVE-2025-38724 use-after-free (CVSS 7.8)**, **CVE-2025-4373 buffer underflow** — all network-reachable, Siemens has not yet shipped fix.

**Pros / Cons vs unidirectional gateway (Waterfall/Owl/Fend) alternative**
- Pros Modbus-direct:
  - Zero CAPEX — user prompt says "new OPC UA historian on Linux VM", no hardware budget mentioned.
  - Preserves UA data model on consumer side — MES team already targets UA.
- Cons Modbus-direct:
  - TCP is bidirectional → write vector exists (protocol-lineage.md line 22: "Unauth by spec; spoof trivial").
  - No cryptographic guarantee of origin or freshness → silent failures #1 and #2 go undetected for hours.
- Pros unidirectional gateway:
  - Hardware-enforced one-way; writes physically impossible; satisfies 62443 zone isolation without custom firewall rules.
  - Gateway typically synthesises proper OPC UA quality + heartbeat → silent failure #1 becomes detectable.
- Cons unidirectional gateway:
  - USD 40–80k per link; adds vendor + config artefact MES must maintain.
  - Backfill/catch-up after gateway reboot needs explicit design; naive deployment loses samples.

**Bottlenecks**
- S7-1500 Modbus TCP server FB is single-connection per instance on older CP modules; historian polls can starve HMI polling.
- DMZ stateful firewall idle-timeout triggers silent failure #1 — commonly 3600 s on Palo Alto/Fortinet defaults.

**Hidden issues & loopholes**
- First-use certificate auto-accept on historian's UA server — rogue UA client on MES VLAN gets trusted forever (protocol-lineage.md line 127).
- Modbus FC01/02 vs FC03/04 off-by-one — 0-based vs 1-based maps tag "40001" to register 0 or 1 depending on gateway (protocol-lineage.md line 21).
- S7-1500 firmware 3.1.5 RCE-class CVEs unpatched as of 2025-06-10 — any TCP exposure to DMZ VM expands reachable attack surface.

**Residual risk**
Even with ACL tightening and read-only Modbus FB code: (a) no cryptographic freshness guarantee, (b) DMZ VM compromise = L1 write access, (c) unpatched Siemens CVEs reachable from closer network position. **Cyber recommendation: do not cross the L3.5→L1 boundary with Modbus TCP.** Either (i) enable native OPC UA on S7-1500 and terminate at DMZ UA aggregator with client certs, or (ii) insert a data diode/unidirectional gateway.
